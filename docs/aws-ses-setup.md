# AWS SES Setup For Cognito OTP

One-time setup for sending Cognito `EMAIL_OTP` emails through Amazon SES from your domain.

This flow is intentionally manual. SES domain verification records are generated per domain, so a dedicated automation script would add little value and would still require human review in DNS.

## What is manual

Run these steps once per sending domain and SES region:

1. Verify one email address in SES.
2. Verify the sending domain in SES.
3. Add the SES DNS records in Cloudflare.
4. Request SES production access for the region.

For this project, use the same region as the Cognito user pool and deployment stack. Current default: `eu-central-1`.

## Domain values for this project

- Website: `https://flashcards-open-source-app.com`
- Repository: `https://github.com/kirill-markin/flashcards-open-source-app`
- Suggested sender: `no-reply@flashcards-open-source-app.com`
- Suggested MAIL FROM domain: `mail.flashcards-open-source-app.com`

## Cloudflare DNS records

SES generates unique DKIM records for each domain identity. Add the exact values shown in the SES console.

When using Cloudflare:

- Keep SES records as `DNS only`.
- Use short names where possible, for example `mail` instead of `mail.flashcards-open-source-app.com`.
- TXT values do not need manual quote escaping beyond what the Cloudflare form requires.

Typical records:

- `3 x CNAME` for DKIM under `*_domainkey`
- `1 x MX` for the custom `MAIL FROM` subdomain
- `1 x TXT` SPF record for the custom `MAIL FROM` subdomain
- `1 x TXT` DMARC record for `_dmarc`

## Leaving the SES sandbox

SES sandbox status is regional. Production access must be requested in the same region as the SES identity and Cognito user pool.

Example status check:

```bash
aws sesv2 get-account --region eu-central-1
```

Example production-access request:

```bash
aws sesv2 put-account-details \
  --region eu-central-1 \
  --production-access-enabled \
  --mail-type TRANSACTIONAL \
  --website-url https://flashcards-open-source-app.com \
  --additional-contact-email-addresses kirill+flashcards@kirill-markin.com \
  --contact-language EN
```

AWS reviews the request manually. This is the normal SES flow and does not require a separate paid support plan.

## Suggested SES production-access description

Use this text if AWS asks for more context:

```text
We are requesting Amazon SES production access in eu-central-1 for transactional email only.

Project:
Flashcards Open Source App
Website: https://flashcards-open-source-app.com
Repository: https://github.com/kirill-markin/flashcards-open-source-app

Use case:
We use Amazon Cognito passwordless email OTP authentication. Amazon SES will be used only to deliver one-time sign-in verification codes to users who explicitly request login by entering their email address in our app.

Email type:
Transactional only. No marketing campaigns, no purchased lists, no bulk promotional email.

Recipient source:
Users provide their own email address directly in the app when they want to sign in. Emails are sent only as a direct response to that user action.

Content:
A one-time login code for authentication. No unsolicited content.

Abuse prevention:
We use rate limits on OTP requests, verification attempt limits, and short-lived OTP sessions. We do not send to scraped or purchased lists.

Expected volume:
Low initial volume during early product development, with gradual growth as the product launches publicly.

This is an open-source project, and the authentication flow is implemented in the public repository linked above.
```

## What belongs in infrastructure code

After SES is verified and production access is approved, keep the repeatable part in CDK:

1. Configure Cognito to use SES with `UserPoolEmail.withSES(...)`.
2. Add a default sender such as `no-reply@flashcards-open-source-app.com`.
3. Add an SES configuration set for delivery, bounce, and complaint events.
4. Wire SES event publishing to the monitoring destination you choose.

The manual SES onboarding above should stay out of CDK because it depends on account review and DNS values generated by AWS at setup time.
